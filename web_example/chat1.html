你可以直接用的自动合并脚本（可选）

如果你不想手工粘贴，可以用一个小脚本把目录下的代码按路径拼成一个 bundle，并自动写 BEGIN/END。

下面是一个简单的 Python 版本（递归收集指定后缀，输出一个 bundle 文本）：

python
import os
from pathlib import Path

ROOT = Path("Assets/Scripts")          # 改成你的根目录
OUT  = Path("bundle_core_ui.md")       # 输出文件名
EXTS = {".cs", ".shader", ".cginc"}    # 你要合并的后缀

def rel(p: Path) -> str:
    return p.as_posix()

files = sorted([p for p in ROOT.rglob("*") if p.suffix in EXTS and p.is_file()],
               key=lambda p: rel(p))

with OUT.open("w", encoding="utf-8") as f:
    f.write(f"PROJECT: (your project)\nBUNDLE: {OUT.name}\n")
    f.write("RULE: Treat each FILE block as original source file.\n\n")

    f.write("FILE INDEX:\n")
    for p in files:
        f.write(f"- {rel(p)}\n")
    f.write("\n")

    for p in files:
        path = rel(p)
        f.write(f"## FILE: {path}\n")
        lang = "csharp" if p.suffix == ".cs" else ""
        f.write(f"```{lang}\n")
        f.write(p.read_text(encoding="utf-8", errors="replace"))
        if not f.tell() or not str(f).endswith("\n"):
            f.write("\n")
        f.write("```\n\n")

print(f"Bundled {len(files)} files -> {OUT}")

方案 1（最推荐）：本机 localhost WebSocket（或 HTTP）桥接

架构

  • VS Code 扩展：启动一个 localhost WebSocket/HTTP server

  • 浏览器扩展：background/service worker 连接 ws://127.0.0.1:<port>，用 JSON 消息交互

  • background 再把消息分发给 content script（或者反过来）

优点

  • 不需要安装额外原生程序（纯 JS/TS）

  • 跨 Chrome/Edge/Firefox 都能做

  • 双向通信简单，吞吐高（适合实时状态/日志/进度）

关键注意点

  1. 认证：不要裸开端口。至少做一次握手 token（随机生成、短时有效）。

  2. CORS：WebSocket 不走 CORS，但你仍应校验 Origin/自定义 header（浏览器端 header 受限时就用首包消息带 token）。

  3. MV3 休眠：Chrome MV3 background 是 service worker，会被挂起。要做断线重连，必要时在 content script 侧维持活动或用 alarms 定期唤醒。

VS Code 扩展端（WebSocket Server）示例

package.json 里依赖：ws

ts
// extension.ts (VS Code)
import * as vscode from "vscode";
import { WebSocketServer } from "ws";
import * as crypto from "crypto";

let wss: WebSocketServer | undefined;

export function activate(ctx: vscode.ExtensionContext) {
  const token = crypto.randomBytes(16).toString("hex"); // 简单握手 token
  const port = 17321;

  wss = new WebSocketServer({ port, host: "127.0.0.1" });

  wss.on("connection", (ws, req) => {
    let authed = false;

    ws.on("message", (raw) => {
      let msg: any;
      try { msg = JSON.parse(raw.toString()); } catch { return; }

      // 首包认证
      if (!authed) {
        if (msg?.type === "auth" && msg?.token === token) {
          authed = true;
          ws.send(JSON.stringify({ type: "auth_ok" }));
        } else {
          ws.close();
        }
        return;
      }

      // 业务消息
      if (msg.type === "ping") {
        ws.send(JSON.stringify({ type: "pong" }));
      } else if (msg.type === "getSelection") {
        const editor = vscode.window.activeTextEditor;
        const text = editor ? editor.document.getText(editor.selection) : "";
        ws.send(JSON.stringify({ type: "selection", text }));
      }
    });
  });

  // 给用户一个可复制的连接信息（你也可以写到状态栏）
  vscode.window.showInformationMessage(
    `Bridge ready: ws://127.0.0.1:${port} token=${token}`
  );

  ctx.subscriptions.push({ dispose: () => wss?.close() });
}

export function deactivate() {
  wss?.close();
}

let ws;
let token = null;   // 你可以让用户在扩展选项页填，或从 VS Code 扩展复制粘贴
let url = "ws://127.0.0.1:17321";

function connect() {
  ws = new WebSocket(url);

  ws.onopen = () => {
    ws.send(JSON.stringify({ type: "auth", token }));
    ws.send(JSON.stringify({ type: "ping" }));
  };

  ws.onmessage = async (ev) => {
    const msg = JSON.parse(ev.data);

    if (msg.type === "selection") {
      // 例：把 selection 发给 content script
      const [tab] = await chrome.tabs.query({ active: true, currentWindow: true });
      if (tab?.id) chrome.tabs.sendMessage(tab.id, msg);
    }
  };

  ws.onclose = () => setTimeout(connect, 1000);
  ws.onerror = () => { try { ws.close(); } catch {} };
}

方案 2：浏览器“Native Messaging”（更强、更复杂）

架构

  • 浏览器扩展 ↔ Native Host（本机可执行程序，通过 stdin/stdout JSON 消息）

  • Native Host 再 ↔ VS Code 扩展（可以是同一个进程、或本机 socket/stdio）

优点

  • 不需要开放 localhost 端口

  • 可访问本机文件系统、进程等（能力强）

缺点

  • 需要安装/注册 native host（Windows 注册表、macOS/ Linux manifest 文件）

  • 分发和权限复杂得多

适合你要做“系统级同步代理”的场景；不适合轻量原型。

方案 3：自定义 URL Scheme（单向触发，适合“召回 VS Code”）

  • 浏览器扩展发起 vscode://... 或自定义协议链接

  • VS Code 扩展通过 vscode.env.asExternalUri/命令处理来接收参数

特点：通常更适合“触发”而不是“持续双向通信”，例如“在 VS Code 打开某文件/跳转到某行”。

方案 4：共享文件/剪贴板（最简单但不可靠）

  • 双方读写同一个临时文件（如 ~/AppData/.../bridge.json）

  • 或借剪贴板 + 用户手动操作

只建议做调试或临时方案，不建议做长期产品。

推荐选型

  • 你要做“稳定双向、实时、跨平台、无需安装额外组件”：方案 1（localhost WebSocket）

  • 你要做“高权限、强集成、尽量少暴露网络面”：方案 2（Native Messaging）

  • 你只要“浏览器一键让 VS Code 执行某动作”：方案 3（URL scheme）

安全与工程化建议（方案 1 的最低标准）

  1. 只监听 127.0.0.1，不要 0.0.0.0

  2. 握手 token + 短期有效（可每次 VS Code 启动变更）

  3. 消息做 schema 校验（至少校验 type/字段）

  4. 断线重连 + 版本协商（hello {version}）

1. **手动 Fetch / Push 的按钮化流程**：用户希望通过按钮完成“从网页取内容 / 向网页推内容”，而不是全自动。
2. **能在 VS Code 内集中查看网页对话片段**：形成开发协作闭环（对话→任务→代码→再对话）。
3. **同步代码可控且结构清楚**：打包后 ChatGPT 能理解“每段内容属于哪个原文件”。
4. **规划/状态机体验接近 Cursor**：Done → Next 连续推进，减少手动切换成本。
5. **ZIP 链接到 VS Code 的闭环**：检测→下载→解压→打开，降低内容导入成本。
6. **不使用 OpenAI API**：与 ChatGPT 网页端的集成通过浏览器侧与用户操作完成。
